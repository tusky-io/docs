API keys give you programmatic access to Tusky through the [REST API](/api-reference/introduction) and the [TypeScript SDK](/sdk/getting-started). Each key carries a set of **scopes** that control exactly which operations it can perform, following the principle of least privilege.

## Key format

Tusky API keys use a prefixed format that indicates the target environment:

| Prefix | Environment | Description |
| --- | --- | --- |
| `tsk_live_` | **Production** | Full access to mainnet resources and real Walrus storage. |
| `tsk_test_` | **Testnet** | Scoped to testnet — safe for development and CI pipelines. |

<Tip>
  Always check the prefix before deploying. Using a testnet key in production (or vice-versa) will result in authentication errors.
</Tip>

## Creating an API key

<Steps>
  <Step title="Open the API keys page">
    Navigate to [app.tusky.io/account/api-keys](https://app.tusky.io/account/api-keys) in the Tusky dashboard.
  </Step>
  <Step title="Click Create API Key">
    Provide a descriptive name (e.g., `production-backend`, `ci-pipeline`, `staging-read-only`) and select the permission scopes your key needs.
  </Step>
  <Step title="Copy and store securely">
    Your API key is displayed **once** after creation. Copy it immediately and store it in a secrets manager or environment variable.

    <Warning>
      You cannot view the full key again after leaving this page. If you lose it, revoke the key and create a new one.
    </Warning>
  </Step>
</Steps>

## Available scopes

Each API key can be assigned any combination of the following scopes:

| Scope | Description |
| --- | --- |
| `vaults:read` | List and retrieve vault metadata |
| `vaults:write` | Create, update, and delete vaults |
| `files:read` | List, retrieve, and download files |
| `files:write` | Upload and delete files |
| `members:read` | List vault members |
| `members:write` | Add, update, and remove vault members |
| `account:read` | Read account info, usage statistics, and wallet balance |
| `aggregators:read` | Read aggregator configuration and analytics |
| `aggregators:write` | Update aggregator settings and custom domains |
| `publisher:read` | Read publisher configuration |
| `publisher:write` | Update publisher settings (epochs, mode, deletable blobs) |

<Note>
  Write scopes implicitly include the corresponding read scope. For example, `files:write` allows both uploading **and** listing/downloading files.
</Note>

## Using your API key

<Tabs>
  <Tab title="SDK">
    Pass the API key when initializing the Tusky client:

    ```typescript
    import Tusky from "@tusky-io/ts-sdk";

    const tusky = new Tusky({
      apiKey: process.env.TUSKY_API_KEY,
    });

    const vaults = await tusky.vault.list();
    console.log("Vaults:", vaults);
    ```
  </Tab>
  <Tab title="cURL">
    Include the key in the `Authorization` header of every request:

    ```bash
    curl https://api.tusky.io/v2/vaults \
      -H "Authorization: Bearer tsk_live_abc123..."
    ```
  </Tab>
  <Tab title="Fetch (JS)">
    ```javascript
    const response = await fetch("https://api.tusky.io/v2/vaults", {
      headers: {
        Authorization: `Bearer ${process.env.TUSKY_API_KEY}`,
      },
    });
    const vaults = await response.json();
    ```
  </Tab>
</Tabs>

## Scope examples

Common scope combinations for different use cases:

<AccordionGroup>
  <Accordion title="Read-only dashboard integration">
    Scopes: `vaults:read`, `files:read`, `account:read`

    ```typescript
    const tusky = new Tusky({ apiKey: process.env.TUSKY_READONLY_KEY });

    const vaults = await tusky.vault.list();
    const usage = await tusky.account.usage();
    ```
  </Accordion>
  <Accordion title="File upload service">
    Scopes: `vaults:read`, `files:write`

    ```typescript
    const tusky = new Tusky({ apiKey: process.env.TUSKY_UPLOAD_KEY });

    const file = await tusky.file.upload("vault_abc123", "./data.csv", {
      name: "dataset.csv",
      epochs: 5,
    });
    ```
  </Accordion>
  <Accordion title="Full admin access">
    Scopes: all scopes

    ```bash
    curl -X POST https://api.tusky.io/v2/vaults \
      -H "Authorization: Bearer tsk_live_admin_key..." \
      -H "Content-Type: application/json" \
      -d '{"name": "production-vault", "encrypted": true}'
    ```
  </Accordion>
</AccordionGroup>

## Best practices

<CardGroup cols={2}>
  <Card title="Least privilege" icon="shield-halved">
    Grant only the scopes each integration needs. A key used solely for downloads should not carry `vaults:write` or `members:write`.
  </Card>
  <Card title="Separate keys per environment" icon="layer-group">
    Create distinct keys for development, staging, and production. Use `tsk_test_` keys for non-production environments.
  </Card>
  <Card title="Rotate regularly" icon="rotate">
    Rotate keys on a schedule and immediately if a key may have been exposed. Create the new key, update your config, then revoke the old key.
  </Card>
  <Card title="Use environment variables" icon="terminal">
    Never hardcode API keys in source code. Store them in `.env` files (excluded from version control) or a secrets manager.
  </Card>
</CardGroup>

<Warning>
  If you suspect a key has been compromised, revoke it immediately from [app.tusky.io/account/api-keys](https://app.tusky.io/account/api-keys). Revocation takes effect instantly — all in-flight requests using the revoked key will fail.
</Warning>

## Key management

### Rotation workflow

<Steps>
  <Step title="Create a new key">
    Generate a new API key with the same scopes as the key you are rotating.
  </Step>
  <Step title="Update your application">
    Replace the old key in your environment variables, secrets manager, or CI/CD configuration.
  </Step>
  <Step title="Verify">
    Confirm the application works correctly with the new key.
  </Step>
  <Step title="Revoke the old key">
    Delete the old key from the dashboard. Revocation is immediate.
  </Step>
</Steps>

<Info>
  Tusky does not enforce automatic key expiration. You are responsible for rotating keys according to your organization's security policy.
</Info>
