Vaults are the top-level containers in Tusky for organizing and managing your files. Think of them as secure folders or cloud storage buckets — each vault groups related files together and provides granular control over encryption, access, and sharing.

## Vault types

Tusky supports two vault types, each suited to different use cases.

<CardGroup cols={2}>
  <Card title="Public vault" icon="globe">
    Files stored in a public vault are uploaded **unencrypted** to Walrus. Anyone with the blob ID or aggregator URL can retrieve them. Ideal for open data, public websites, and shared media.
  </Card>
  <Card title="Encrypted vault" icon="lock">
    Files are **encrypted client-side** before leaving your device. Only vault members with the correct decryption key — derived from their connected Sui wallet or managed through Seal — can read the contents.
  </Card>
</CardGroup>

<Warning>
  A vault's encryption setting is permanent. You cannot convert a public vault to an encrypted vault (or vice-versa) after creation.
</Warning>

## Creating a vault

<Steps>
  <Step title="Choose a name and type">
    Give your vault a descriptive name and select either **Public** or **Encrypted**. This choice is immutable.
  </Step>
  <Step title="Configure settings">
    Set default storage duration (in epochs), enable auto-extend if desired, and optionally add a description or tags.
  </Step>
  <Step title="Start uploading">
    Once the vault is created you can immediately upload files, add members, or connect the vault to an aggregator.
  </Step>
</Steps>

## Vault settings

Every vault exposes a settings panel where you can manage:

| Setting | Description |
|---|---|
| **Name & description** | Human-readable metadata for the vault. |
| **Default storage epochs** | The number of Walrus storage epochs applied to new uploads unless overridden per-file. |
| **Auto-extend** | When enabled, Tusky automatically renews storage before expiry so your data is never lost. |
| **Encryption** | Displays the encryption status (read-only after creation). |

## Members and roles

Vaults support collaborative access through a role-based membership model.

<Tabs>
  <Tab title="Owner">
    Full control over the vault — can upload, download, delete files, manage members, and modify settings. Each vault has exactly one owner (the creator).
  </Tab>
  <Tab title="Manager">
    Can upload, download, and delete files and invite or remove Viewers. Cannot change vault settings or remove other Managers.
  </Tab>
  <Tab title="Viewer">
    Read-only access — can download and view files but cannot upload, delete, or manage membership.
  </Tab>
</Tabs>

<Info>
  For encrypted vaults, adding a member shares the vault's decryption capability with that member's Sui wallet through Seal. Members never see raw encryption keys.
</Info>

## Vault lifecycle

Understanding the full lifecycle of a vault helps you plan storage and collaboration effectively.

```
Create → Configure → Upload files → Share / add members → Manage → Archive / Delete
```

1. **Create** — A new vault is provisioned and linked to your account.
2. **Active** — Files are uploaded, accessed, and managed. Members collaborate as needed.
3. **Archive** *(optional)* — You can stop new uploads while keeping existing files accessible for their remaining storage duration.
4. **Delete** — Removing a vault deletes all Tusky metadata. Files already published to Walrus remain available until their storage epochs expire; however, they will no longer be served through your Tusky aggregator.

<Note>
  Deleting a vault does **not** immediately delete data from the Walrus network. Walrus blobs persist until their purchased storage epochs expire. If you have ejected blobs to your own wallet, those blobs remain fully under your control regardless of vault status.
</Note>
