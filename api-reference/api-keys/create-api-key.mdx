---
title: Create API Key
api: 'POST https://api.tusky.io/v2/api-keys'
---

Create a new API key. Keys can be **admin** (full access) or **scoped** (restricted to specific operations).

## Request Body

<ParamField body="name" type="string" required>
  A descriptive name for the key (e.g., `production-backend`, `ci-pipeline`).
</ParamField>

<ParamField body="scopes" type="array">
  Permission scopes to grant. Omit (or pass `null`) to create an **admin key** with full access to all operations. See [scopes reference](#available-scopes) below.
</ParamField>

<ParamField body="expiresAt" type="string">
  Optional ISO 8601 expiration date. Key is valid indefinitely if omitted.
</ParamField>

## Available Scopes

| Scope | Description |
| --- | --- |
| `volumes:read` | List and retrieve volumes |
| `volumes:write` | Create, update, delete volumes |
| `files:read` | List, retrieve, and download files |
| `files:write` | Upload and delete files |
| `members:read` | List volume members |
| `members:write` | Add, update, remove members |
| `account:read` | Read account info and usage |

## Use Cases

### Admin key (full access)

Omit `scopes` entirely to create an admin key. Admin keys have unrestricted access to all API operations — equivalent to acting as the account owner.

```json
{
  "name": "admin"
}
```

<Warning>
  Use admin keys only in trusted, server-side environments. Never expose them in client-side code or share them with third parties.
</Warning>

### Frontend uploads

When users upload files directly from your frontend (e.g. a web app or NFT minting page), generate a short-lived key scoped only to writing files. Never expose a full-access key in client-side code.

```json
{
  "name": "frontend-upload-session",
  "scopes": ["files:write", "volumes:read"],
  "expiresAt": "2026-03-15T12:00:00Z"
}
```

<Tip>
  Generate this key server-side per authenticated user session and return it to the client. Set a short `expiresAt` — minutes to hours, not days.
</Tip>

### Read-only access

For frontends that only need to list or display files — dashboards, media galleries, content sites — a read-only key can be used safely in client-side code since it cannot modify any data.

```json
{
  "name": "public-frontend",
  "scopes": ["files:read", "volumes:read"]
}
```

### Backend server

Server-side applications typically need to read and write files. Scope tightly even here — if your backend never manages members, don't include member scopes.

```json
{
  "name": "production-backend",
  "scopes": ["files:read", "files:write", "volumes:read"]
}
```

### Third-party or CI access

When granting access to an external collaborator, agency, or CI pipeline, issue a dedicated key with only the operations they need. This limits exposure if the key is ever compromised.

```json
{
  "name": "ci-pipeline",
  "scopes": ["files:write", "volumes:read"]
}
```

<Tip>
  Set an `expiresAt` for any key issued to an external party. Rotate regularly.
</Tip>

## Response

<ResponseField name="id" type="string">API key ID.</ResponseField>
<ResponseField name="key" type="string">The API key value. **Only returned once** — store it securely.</ResponseField>
<ResponseField name="name" type="string">Key name.</ResponseField>
<ResponseField name="scopes" type="array">Granted scopes.</ResponseField>
<ResponseField name="expiresAt" type="string">Expiration timestamp, or `null`.</ResponseField>
<ResponseField name="createdAt" type="string">ISO 8601 creation timestamp.</ResponseField>

## Example

<CodeGroup>

```bash cURL
curl -X POST https://api.tusky.io/v2/api-keys \
  -H "Api-Key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "production-backend",
    "scopes": ["files:read", "files:write", "volumes:read"]
  }'
```

</CodeGroup>

### Response

```json
{
  "id": "key_abc123",
  "key": "tsk_live_xK9m2...",
  "name": "production-backend",
  "scopes": ["files:read", "files:write", "volumes:read"],
  "expiresAt": null,
  "createdAt": "2026-03-07T14:00:00Z"
}
```

<Warning>
  The `key` value is only returned in this response. Store it securely — you cannot retrieve it again.
</Warning>
